The branch filter `<select>` is placed inside the **page header area** (above the split pane),
positioned left of the "Pipelines" h1 group — or as the first row of a `.filter-toolbar` strip

toggle). If no findings/artifacts are loaded for a run row in the list, tool tags and severity
bar are omitted (zero-findings case: show only the existing 3 rows).

### Trend chart placement (PIPE-04)

A trend card is inserted at the top of the **detail panel** above the `RunPanel` content, ONLY
when the runs list has >= 2 items. The card uses `.card` + `.card-header` + `.card-pad`.

Chart component: `AreaTrend` from `Charts.tsx`.
- `values`: array of failed-run counts per run (last 30 runs, index 0 = oldest)
- `values2`: array of passed-run counts per run (dashed line, `var(--fg-4)` stroke — existing)
- `height`: 120 (compact — override the default 200px)
- Card title: "Pipeline Trend" (`.h3` weight 600 at `--ts-base`)
- Card subtitle: "Pass / fail over last 30 runs" (`.muted` at `--ts-sm`)
- Chart width: 100% of card content area

### Auto-refresh indicator (PIPE-03)

A live-status indicator is shown in the page header right side, adjacent to the Refresh Runs button.
It consists of:
- A 6px circle filled with `var(--accent)`, pulsing via the existing `@keyframes pulse` animation
- Text "Live" at `--ts-xs` weight 600, color `var(--fg-3)`
- The indicator is ONLY visible when `runs.some(r => r.status === 'in_progress')` is true
- When no in-progress runs exist, the indicator is hidden (`display: none`)

The existing 30-second `setInterval` polling in Pipelines.tsx is the refresh mechanism. No
WebSocket or SSE is introduced in Phase 02.

## Interaction Contract

### Branch filter (PIPE-02)

| Trigger | Response |
|---------|----------|
| User selects branch from `<select>` | `runs` list filtered to matching `head_branch`; run count in sub-heading updates; if selected run is no longer in filtered list, auto-select first run in filtered results |
| User selects "All branches" (default option) | Full unfiltered list restored |
| Branch list | Derived from `[...new Set(runs.map(r => r.head_branch))]` — no separate API call |

### Run row click (existing behavior — do not change)

| Trigger | Response |
|---------|----------|
| Click run row | Sets `selectedId`, highlights row with `active` class + `--accent-tint` bg, loads `RunPanel` for that run |

### Refresh Runs button (existing behavior — do not change)

| Trigger | Response |
|---------|----------|
| Click "Refresh Runs" | Calls `refresh()`, clears stale list, re-fetches all runs, re-auto-selects latest CI run |

### Auto-refresh (existing — do not change the interval)

| Event | Response |
|-------|----------|
| 30s interval fires | `api.github.runs()` called silently; `setRuns(arr)` updates list; `LiveIndicator` visibility derived from updated array |
| In-progress run completes | Status chip updates from `status-running` to `status-passed` or `status-failed` on next poll |

### Reprocess Run action (existing — preserve as-is)

| Trigger | Response |
|---------|----------|
| Click "Reprocess Run" button | `window.confirm()` prompt (existing — do NOT replace with AlertBanner; it is a destructive action requiring positive confirmation) |
| Confirm | Sets `reprocessing: true`, shows inline accent-tint info div (existing), loads findings after 10s |

## Implementation Notes for Executor

1. **Do not rename or restructure existing functions** (`RunPanel`, `SeverityBoard`, `ToolBreakdown`,
   `TopFindings`, `SectionHeader`, `renderRunRow`, `conclusionClass`, `conclusionLabel`, `timeAgo`).
   Extend them — do not rewrite.

2. **Branch filter state**: Add `const [branch, setBranch] = useState<string>('all')`. Derive
   `filteredRuns` via `useMemo`. Replace all uses of `runs` in the render tree with `filteredRuns`
   for the list pane. Keep `runs` (unfiltered) for trend chart data and stats KPIs.

3. **Trend data shape**: The trend chart shows pass/fail counts bucketed across the last 30 runs
   in chronological order (index 0 = oldest). Compute from `runs.slice(-30).reverse()` or sort
   by `created_at` ascending. Each point: `{ passed: N, failed: N }`. Pass `values` (failed
   counts) and `values2` (passed counts) to `AreaTrend`.

4. **Duration display**: `WorkflowRun` may not have a `duration_ms` field. If absent, compute
   from `created_at` to `updated_at` (both ISO strings already on the type). Format as
   `Xm Ys` for values under an hour, `Xh Ym` for longer runs. Use `var(--font-mono)`.

5. **Tool tags in run row**: The tools list must come from artifacts or findings already loaded.
   For the sidebar run list, fetch artifacts eagerly when the page loads (or display tool tags
   only for the selected run). If per-row tool data is not available without an extra API call
   per row, show tool tags only for the selected run's row (the `RunPanel` already loads
   artifacts — mirror that data back up). Clarify with backend plan 02-01 which endpoint
   returns tool metadata per run in the list response.

6. **AlertBanner usage**: Use `AlertBanner` (type "error") for the fetch-error state in the
   run list pane. Do NOT use for loading states — use `.empty` text only.

7. **Preserve existing poll interval**: The existing `setInterval(30_000)` in `useEffect` must
   not be modified. The `LiveIndicator` derives its visibility from the `runs` state array.

8. **No new CSS files**: All styles must use existing token variables and class names from
   `tokens.css`. Inline styles using token variables (e.g., `style={{ color: 'var(--fg-3)' }}`)
   follow the existing pattern in Pipelines.tsx and are acceptable.

9. **Font weight rule**: Use only `fontWeight: 400` (regular) or `fontWeight: 600` (emphasis).
   Do not introduce 500 or 700 in any new code. Any existing 500/700 uses encountered during
   Phase 02 work should be corrected to 600.

10. **KPI counter**: match existing inline style `fontSize: 18` in Pipelines.tsx — do not change the value, do not add a new token. This is an inherited pattern.
