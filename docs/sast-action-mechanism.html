sast-action — Cơ chế chạy

GitHub composite Action ở repo target, và contract nó phải khớp với chat-system

Module liên quan: mcp/ (backend contract) · SAST/SCA DevSecOps · Tài liệu kỹ thuật nội bộ

Phạm vi tài liệu. sast-action là một GitHub composite Action nằm ở repo target (theo quy ước ALOUTE_Spring_Thymeleaf_RCE) — không nằm trong repo chat-system này. Tài liệu mô tả cơ chế chạy dựa trên: (a) 3 composite được ghi trong comment của repo này (config/profiles/github-actions-default.yml, api/findings.py); (b) contract mà action buộc phải khớp để chat-system nhận diện được — tên artifact, payload webhook, query gate. Phần YAML nội bộ của action nằm ở repo target.
TL;DR — sast-action là client mỏng & stateless. Nó chỉ làm 3 việc: (1) chạy scanner & upload artifact đúng tên; (2) bắn webhook báo run xong; (3) hỏi gate pass/fail rồi cho CI xanh/đỏ. Mọi "trí tuệ" — normalize, dedup, chính sách threshold — nằm ở server (chat-system), không nằm trong action. Nhờ vậy đổi ngưỡng gate trong dashboard không cần redeploy action.

1. Vị trí & vai trò — 2 repo

Một bên là repo target chứa sast-action (chạy trong GitHub Actions). Bên kia là chat-system (MCP Gateway / FastAPI + DB + dashboard). Chúng nối nhau qua Artifact API + Webhook + Gate API.

2. Ba composite của sast-action

sast-suite

Chạy SAST/SCA song song

Chạy các scanner tĩnh song song: Semgrep, CodeQL, ESLint (SARIF), SpotBugs, OWASP Dependency-Check, Trivy (filesystem + container image).

Kết quả upload thành: artifact gộp sast-reports-<run> và các artifact per-tool (semgrep-report, codeql-report, …). → Chính sự tồn tại song song của bản gộp + per-tool là nguồn gây trùng mà Tầng 3 dedup xử lý ở phía server.

profiles/github-actions-default.yml (comment "sast-reports-* = bundled multi-tool từ sast-suite")

run-dast

Quét động (DAST) — OWASP ZAP

Quét app đang chạy bằng OWASP ZAP (V2.3), upload dast-reports-<run>. Webhook DAST land riêng nhưng gọi process_run với cùng run_id → idempotency guard ở server tránh nhân đôi.

profiles/github-actions-default.yml · processor.py:129-133

security-gate

Cổng chặn merge

Sau khi scan xong: (1) bắn webhook báo run hoàn tất; (2) poll gate-count tới khi run_ingested=true; (3) đọc pass do server tính theo threshold của project; (4) pass=false → fail job, chặn merge.

api/findings.py:185 · vì sao gate ở server: docs/fix_bugs.md:373

5. Vì sao gate verdict tính ở server, không ở action

  • CI runner stateless: action không giữ dữ liệu finding, không biết lịch sử triage/approve/revoke. Chỉ server mới biết finding nào đã bị REVOKE/APPROVE để loại khỏi đếm.
  • Đổi policy không cần redeploy: threshold nằm trong DB per-project; owner chỉnh qua PATCH /projects/{id}/gate-policy mà không phải sửa YAML rồi phát hành lại sast-action. docs/fix_bugs.md:373
  • Một nguồn sự thật: gate, dashboard KPI, list, AI summary đều đọc cùng số đã dedup ở server → CI ↔ dashboard luôn khớp.
