Flow /findings/gate-count

Endpoint quyết định pass/fail của Security Gate — CI poll để cho merge hay chặn

File: mcp/src/api/findings.py:185 · V3.6 (verdict server-side) + V3.7 (run_ingested) · Tài liệu kỹ thuật nội bộ

TL;DR — gate-count là một câu hỏi mà CI security-gate lặp lại tới khi ingest xong: "run này còn bao nhiêu lỗi ACTIVE, và có vượt ngưỡng không?". Server đếm finding (loại REVOKED + APPROVED), so với threshold của project, rồi trả pass: bool. Quyết định nằm ở server (không ở CI) nên đổi ngưỡng trong dashboard không cần sửa CI YAML.

4. Ngữ nghĩa đếm (SQL thật)

Mỗi lần đếm 1 severity, count_with_filters sinh câu:

SELECT count(findings.id)
FROM findings JOIN artifacts ON findings.artifact_id = artifacts.id
WHERE artifacts.project_id   = :project_id
  AND artifacts.github_run_id = :run_id       -- khóa nối với CI
  AND findings.severity        = :severity
  AND findings.status <> 'REVOKED'             -- exclude_revoked
  AND findings.status <> 'APPROVED'            -- exclude_approved

Trạng thái findingCó tính vào gate?Vì sao

Liên hệ với dedup: run_id ở đây chính là khóa map CI ↔ dashboard. Vì Tầng 3 dedup đã gộp trùng trong run trước khi run_ingested bật, nên số gate đọc được = số dashboard hiển thị (cùng tập row đã khử trùng). Chi tiết: docs/dedup-ci-local-flow.html.

5. Chính sách threshold

Field (per-project)Mặc địnhÝ nghĩa
gate_critical_threshold0Số critical tối thiểu để CHẶN
gate_high_threshold5Số high tối thiểu để CHẶN
⚠ Footgun: threshold = 0 nghĩa là TẮT gate cho severity đó — KHÔNG phải "cấm tuyệt đối". Điều kiện chặn là crit_thr > 0 AND critical ≥ crit_thr findings.py:262. Nếu crit_thr = 0 thì vế đầu là false → critical không bao giờ chặn. Vậy với mặc định (critical=0, high=5): critical không gate, còn high chặn khi có ≥ 5. Muốn "chặn nếu có bất kỳ critical nào" → đặt gate_critical_threshold = 1 qua PATCH /projects/{id}/gate-policy (chỉ owner/admin). projects.py:471

  "exclude_approved": true,
  "critical": 0,

  "medium": 41,

  "run_ingested": true,
  "policy": { "critical_threshold": 1, "high_threshold": 5 },
  "pass": true,              // high=3 < 5 và critical=0 < 1 → không có lý do chặn
  "blocking_reasons": []
}

Khi pass=false, blocking_reasons liệt kê dạng "high findings 7 >= threshold 5" để CI in ra log.
